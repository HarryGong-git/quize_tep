# 四章章节要点

## Chapter 1: Database, OLTP, Relational Model, ERD, Normalization

### 1.1 课程主线

这门课的主线是从业务数据到商业智能：

业务过程产生数据 → OLTP 数据库保存交易数据 → 用 SQL 查询和处理数据 → 为分析建立 Data Warehouse → 用 ETL 把源系统数据抽取、清洗、转换、加载到 DW → 支持 BI dashboard 和决策分析。

### 1.2 OLTP 与数据库基础

OLTP, Online Transaction Processing, 指支持日常业务交易的数据系统，例如开户、支付、ATM 取款、订单提交。

OLTP 的特征：

- 数据来源是 operational data，是原始交易数据。
- 目标是支持业务运行，而不是复杂分析。
- 查询通常简单、标准化，返回少量记录。
- 插入和更新频繁，要求速度快。
- 数据库设计通常高度 normalized，有很多表。
- 需要严格 backup and recovery，因为数据丢失会影响业务和法律责任。

事务 transaction 的核心：

- 一个业务动作通常包含多条数据库操作。
- 这些操作要作为一个 unit of work 被可靠执行。
- 成功则 `COMMIT`，失败则 `ROLLBACK`。
- DBMS 需要处理并发控制和故障恢复。

### 1.3 Relational Model

关系模型把数据库看成一组 tables。

核心术语：

- Relation / Table：表。
- Tuple / Row / Record：行。
- Attribute / Column / Field：列。
- Heading：表名和列名。
- Body：实际数据行。
- Query result：查询结果本身也可以看成一张表。

三类完整性约束：

- Entity integrity：每张表有唯一且非空的 primary key。
- Referential integrity：foreign key 必须引用源表中存在的 key。
- Domain integrity：列值必须满足类型、格式、范围、枚举等限制。
- User-defined integrity：业务规则，例如取款金额不能超过余额。

Primary Key:

- 唯一标识一行记录。
- 不能重复，不能为 null。
- 可以是一列，也可以是多列组合。

Foreign Key:

- 一个表中的列，引用另一个表的 primary key 或 unique key。
- 用来表达表之间的关系。
- 保证引用有效，避免孤立或无效记录。

### 1.4 ERD 数据库设计

ERD, Entity-Relationship Diagram, 用于描述业务领域中的实体和关系。

常见符号：

- Rectangle：entity。
- Diamond：relationship。
- Oval：attribute。
- Underline：key。

ERD 的作用：

- 设计新数据库。
- 理解已有数据库中表之间的连接。
- 作为沟通工具，让业务和技术对数据结构达成一致。

从 ERD 转为 tables 的步骤：

1. 每个 entity 建一张表。
2. 为每张表确定 primary key。
3. 为 attributes 选择数据类型。
4. 对复杂属性拆分，例如 name、phone numbers。
5. 对 relationship 建表或放 foreign key。
6. 1:N 关系通常在 N 端放 foreign key。
7. M:N 关系通常建 junction / bridge table，使用两边 foreign keys。
8. 添加完整性约束，例如 `NOT NULL`、`UNIQUE`、`CHECK`、`FOREIGN KEY`。

### 1.5 Normalization

Normalization 的目标：

- 减少 data redundancy。
- 提高 data integrity。
- 让非 key 属性紧密依赖 primary key。

1NF:

- 每个 cell 只包含一个 atomic value。
- 每行唯一。
- 不能在一个单元格里放多个产品、多个电话等重复组。

2NF:

- 消除 partial dependency。
- 非 key 属性必须依赖整个 composite key，而不是只依赖其中一部分。
- 如果主键是单列，通常天然满足 2NF。

3NF:

- 消除 transitive dependency。
- 非 key 属性只能依赖 primary key，不能依赖另一个非 key 属性。
- 例如 `CID -> City` 且 `City -> Country`，则 `Country` 应拆到 `City` 表。

Normalization 与后续 DW 的联系：

- OLTP 喜欢 normalized design，因为更新快、冗余少、数据一致性好。
- Analytics 不喜欢过度 normalized design，因为分析查询要 join 很多表，速度慢、写 SQL 复杂。
- 这就是 Chapter 3 要引入 Data Warehouse 和 dimensional model 的原因。

## Chapter 2: SQL

### 2.1 SQL 语句类型

常见 SQL statements：

- DDL: `CREATE TABLE`, `CREATE VIEW`, `CREATE TYPE`。
- DML: `SELECT`, `INSERT`, `UPDATE`, `DELETE`。
- Transaction control: `COMMIT`, `ROLLBACK`。
- Access control: `GRANT`, `REVOKE`。

常见数据类型：

- Text: `CHAR(L)`, `VARCHAR(L)`。
- Number: `INTEGER`, `FLOAT`, `DECIMAL(W,R)`。
- Date/time: `DATE`, `TIME`, `TIMESTAMP`。
- Boolean: `BOOLEAN`。
- Binary: `BLOB`。

钱、金额、精确小数优先用 `DECIMAL`，不要用 `FLOAT`，因为 floating point 可能产生 rounding error。

### 2.2 CREATE TABLE

基本结构：

```sql
CREATE TABLE table_name (
  column_name data_type constraint,
  ...
  CONSTRAINT constraint_name PRIMARY KEY (column_name),
  CONSTRAINT constraint_name UNIQUE (column_name),
  CONSTRAINT constraint_name FOREIGN KEY (fk_column) REFERENCES parent_table(parent_key),
  CONSTRAINT constraint_name CHECK (condition)
);
```

写 `CREATE TABLE` 时要考虑：

- 表名和列名。
- 每列数据类型。
- 哪列是 primary key。
- 哪些列不能为 null。
- 哪些列必须 unique。
- 哪些列只能取固定值。
- 是否有 foreign key。

### 2.3 SELECT 基础

标准结构：

```sql
SELECT column_expressions
FROM tables_or_joins
WHERE row_conditions
ORDER BY sort_columns;
```

常见考点：

- `SELECT *` vs 指定列。
- `WHERE` 过滤行。
- `AND`, `OR`, `NOT` 组合条件。
- `LIKE` 做文本匹配，`%` 表示任意长度，`_` 表示单个字符。
- `BETWEEN` 做日期或数值范围。
- `IS NULL` / `IS NOT NULL` 判断空值。
- `DISTINCT` 去重。
- `ORDER BY ... ASC/DESC` 排序。
- `LIMIT n` 取 top n。

### 2.4 JOIN

JOIN 用来把多个表按相关列合并，最常见条件是 `PK = FK`。

常见类型：

- `INNER JOIN`：只保留匹配行。
- `LEFT JOIN`：保留左表全部行，右表匹配不到则为 null。
- `RIGHT JOIN`：保留右表全部行。
- `FULL OUTER JOIN`：保留两边全部行，MySQL 不直接支持，通常用 `LEFT JOIN UNION RIGHT JOIN` 模拟。
- `CROSS JOIN`：笛卡尔积，没有 join condition，通常危险。
- `SELF JOIN`：表自己 join 自己。

推荐写法：

```sql
SELECT ...
FROM table1 t1
JOIN table2 t2 ON t1.key = t2.key
WHERE ...
```

不要优先用 natural join，因为它依赖同名列隐式连接，容易不可控。

### 2.5 Aggregation

Aggregation 是把多行压缩为每组一个 summary。

常见函数：

- `COUNT(*)`
- `COUNT(DISTINCT column)`
- `SUM(column)`
- `AVG(column)`
- `MIN(column)`
- `MAX(column)`

关键规则：

- `GROUP BY` 定义分组粒度。
- `SELECT` 中非聚合列必须出现在 `GROUP BY`。
- `WHERE` 过滤 individual rows，在分组前发生。
- `HAVING` 过滤 groups，在分组后发生。

常见模板：

```sql
SELECT group_col, COUNT(*) AS cnt, SUM(amount) AS total_amount
FROM table_name
WHERE row_condition
GROUP BY group_col
HAVING COUNT(*) > 10
ORDER BY total_amount DESC;
```

### 2.6 More SQL: Set Operators, IF/CASE, Functions

Set operators 处理 rows，不是 columns：

- `UNION`：合并并去重。
- `UNION ALL`：合并但保留重复，速度更快。
- `INTERSECT`：取交集，MySQL 支持情况取决于版本。
- `MINUS` / `EXCEPT`：取差集，不同数据库写法不同。

Set operators 要求：

- 两边 query 列数相同。
- 对应位置的数据类型兼容。
- 按位置匹配，不按列名匹配。

`IF` 和 `CASE`：

```sql
SELECT IF(quantity > 10, 'High', 'Low') AS quantity_level
FROM order_details;
```

```sql
SELECT
  CASE
    WHEN quantity > 30 THEN 'greater than 30'
    WHEN quantity = 30 THEN 'equal to 30'
    ELSE 'under 30'
  END AS quantity_group
FROM order_details;
```

常见 MySQL 函数：

- String: `CONCAT`, `LOWER`, `UPPER`, `SUBSTRING`, `TRIM`。
- Date/time: `CURDATE`, `DATE`, `YEAR`, `MONTH`, `DATEDIFF`, `TIMESTAMPDIFF`, `DATE_ADD`。
- Number: `ROUND`, `ABS`, `SUM`, `AVG`, `MAX`, `MIN`。
- Null: `IFNULL`, `COALESCE`, `NULLIF`。

### 2.7 Subquery 与 CTE

Subquery 是嵌套在另一个 SQL 中的 query，可出现在 `SELECT`, `FROM`, `WHERE`, `HAVING`。

例子：找人口高于平均人口的国家。

```sql
SELECT name, population
FROM country
WHERE population > (SELECT AVG(population) FROM country);
```

CTE, Common Table Expression, 用 `WITH` 定义临时命名结果集。

```sql
WITH avg_pop AS (
  SELECT AVG(population) AS avg_p
  FROM country
)
SELECT c.name, c.population
FROM country c, avg_pop a
WHERE c.population > a.avg_p;
```

CTE 优点：

- 可读性强。
- 能把复杂逻辑拆成步骤。
- 同一个 CTE 可在主查询中复用。
- 适合复杂报表和多步查询。

## Chapter 3: Data Warehouse and Dimensional Model

### 3.1 为什么需要 Data Warehouse

如果直接在 OLTP 上做 BI dashboard：

优点：

- 架构简单。
- 能看到最新数据。
- 不需要额外建设 DW。

问题：

- OLTP 必须保留大量历史数据。
- 复杂分析查询可能拖垮业务系统。
- 分析查询需要 join 很多 normalized tables。
- 历史数据多、关系静态，join 代价很高。

Data Warehouse 的思想：

- 把不同 source 的数据提前整合。
- 存入专门为查询和分析优化的仓库。
- 让复杂查询在 DW 上执行，不干扰 OLTP。

### 3.2 DW 的特征

Data Warehouse 是：

- Stored collection of diverse data。
- Single repository of information。
- Subject-oriented，按业务主题组织，不按应用系统组织。
- Optimized for analysis，而不是 transaction。
- Large volume，通常 GB/TB。
- Non-volatile，保存历史，更新不频繁。
- Time attributes are important。
- Often append-only。

OLTP vs OLAP：

| 方面 | OLTP | OLAP / Warehouse |
|---|---|---|
| 工作负载 | mostly updates | mostly reads |
| 设计 | normalized | denormalized |
| 查询 | small/simple transactions | long/complex queries |
| 数据量 | MB-GB | GB-TB |
| 时间 | current snapshot | full history |
| 数据 | raw data | summarized/reconciled data |
| 用户 | app users/clerical users | analysts/decision-makers |

Data Mart:

- Data warehouse 的子集。
- 面向特定业务线或部门，例如 HR、Finance、Marketing。
- 优点是上线快，缺点是长期集成更复杂。

### 3.3 Dimensional Model

Dimensional modeling 用业务概念组织数据。

ER model 描述 entities and relationships。

Dimensional model 描述 measures and dimensions。

Measures:

- 被跟踪的数值型业务指标。
- 通常可加总，例如 sales amount、quantity、number of rentals。

Dimensions:

- 描述交易上下文的业务参数。
- 相对静态，例如 time、customer、product、store、merchant category。
- 用来过滤、分组、切片、展示报表标签。

典型问题：

- Sales by product line over the past six months。
- Sales by store between 2010 and 2015。
- Transaction amount by merchant category and month。

### 3.4 Star Schema 与 Snowflake Schema

Star schema:

- 一个 fact table 在中心。
- 多个 dimension tables 围绕 fact table。
- Fact table 保存 measures 和指向 dimensions 的 foreign keys。
- Dimension tables 通常 denormalized。

优点：

- 易理解。
- 业务实体映射直观。
- join 少，查询性能好。
- metadata 简单。
- 适合 end users 浏览和 OLAP。

缺点：

- dimension tables 可能很大。
- 如果把 summary data 混在 fact table，可能影响 summary level 性能。

Snowflake schema:

- 把 dimension tables 进一步 normalized。
- 例如 product dimension 拆成 product、brand、category。

优点：

- 减少冗余和存储。
- dimension 更新维护更容易。

缺点：

- 查询更复杂。
- join 更多，性能可能更差。
- end users 不容易理解。

考试建议：如果问 DW 常规推荐设计，优先答 star schema；snowflake 只有在 dimension 很大、维护压力大、需要减少冗余时考虑。

### 3.5 Grain

Grain 是 fact table 每一行代表的业务细节层级。

定义 grain 是 dimensional model 最重要的步骤。

例子：

- 一张 grocery receipt 上的一条 line item。
- 一个 invoice 上的单个 item。
- 某天购买的一张 airline ticket。
- 仓库库存的 weekly snapshot。
- 银行账户 monthly snapshot。

设计原则：

- 先明确 fact table 一行是什么意思。
- Measures 必须和 grain 一致。
- 不同 grain 最好用不同 fact tables。
- 不推荐在一个 fact table 里混 daily/monthly/yearly，再用 granularity flag 区分。

### 3.6 Dimension Table 设计

Surrogate key:

- DW 自己生成和维护的 artificial key。
- 推荐作为 dimension table 的 primary key。
- 不直接使用 source system 的 natural key。

为什么使用 surrogate key：

- 不同 OLTP source 可能对同一实体使用不同 key。
- 一个 source key 可能被重复或复用。
- 合并多个 source 时 natural key 不稳定。
- 支持维度变化和历史追踪。

Dimension attributes:

- 非 key 列一般称为 attributes。
- 应该是业务描述性列，方便报表过滤、分组和显示。
- 如果有 code，通常也要保存 description。
- 报表字段要对业务用户友好，例如用 Yes/No 而不是 0/1。

Date dimension:

- 几乎所有 DW 都需要 date dimension。
- 通常可以独立生成。
- 包含 date、day、month、quarter、year、fiscal year、holiday 等。

Time dimension:

- 通常与 date dimension 分开。
- 如果按秒建在 date dimension 中，一年会有 31,536,000 行。
- 粒度根据用户需求决定，例如 hour、hour range。

Not applicable row:

- Fact table 的 foreign key 不应为 null。
- 维度未知或不适用时，用特殊维度行表示。
- 例如 `Non Member Customer`, `Date unknown`, `Does not have insurance`。

Degenerate dimension:

- 没有 attributes 的 dimension。
- 通常是 transaction number、invoice number、order number。
- 直接放在 fact table 中。
- 多用于 fine-grained fact table。

Conformed dimension:

- 被多个 fact tables 共享或可能共享的 dimension。
- 用来保证跨业务主题分析时口径一致。

Slowly Changing Dimension:

- 维度属性随时间缓慢变化，例如 customer address、customer segment。
- Type 1：覆盖旧值，不保留历史。
- Type 2：新增一行，使用新的 surrogate key，保留历史。
- Type 3：增加 previous value 列，记录有限历史。

Fast Changing Dimension:

- 属性变化很快，不能用普通 SCD Type 1/2/3 高效处理。
- 通常把快变属性拆到 mini-dimension。

## Chapter 4: ETL

### 4.1 ETL 定义

ETL = Extract, Transform, Load。

- Extract：从 source systems 抽取相关数据。
- Transform：清洗、转换、整合、去重、类型转换、生成 key、聚合。
- Load：把转换后的数据加载进 data warehouse。

ETL 的目标是把源系统数据变成 DW/BI 可以高效查询、语义清晰、质量可控的数据。

### 4.2 Data Staging Area

Data Staging Area, DSA, 是 ETL 过程中的中转存储区域。

作用：

- 暂存从 source 抽取的数据。
- 在这里执行 cleaning 和 transformation。
- 支持分步骤处理，便于重启。
- 方便记录 lineage 和中间结果。
- 避免所有转换一步完成导致难 debug、难恢复。

为什么需要 DSA：

- ETL 常处理大数据量，必须分阶段。
- 源系统和 DW 不应被复杂转换过程互相拖累。
- 出错时可以从某个 staging step 重新开始。
- 可以保存抽取快照，满足审计、追踪、质量检查。

DSA 通常不面向用户查询。

### 4.3 ETL 设计约束

ETL 设计会受到很多现实约束：

- Business needs: KPI 和业务需求。
- Compliance: 监管要求。
- Data quality: 准确性、完整性、一致性。
- Security: 效率与安全平衡。
- Data integration: conformed dimensions and conformed facts。
- Data latency: 数据需要多快到达用户。
- Archiving and lineage: 中间结果和来源追踪。
- BI delivery interfaces: 数据结构要让 BI 简单、快速。
- Available skills: 手写 ETL 还是用工具。
- Legacy licenses: 现有系统和授权限制。

### 4.4 Extract

Extract 的目标：快速抽取 relevant data。

方式：

- SQL-based extract。
- DB unload tools，通常更快。
- Extract applications，在某些场景是唯一方法。

关键思想：

- 不要每次都全量抽取。
- 大系统中全量抽取可能需要几天或几周。
- 更常见是抽取 delta，即自上次加载以来发生变化的数据。

### 4.5 Transform

常见 transformation：

- 数据类型转换。
- 编码转换，例如 EBCDIC 到 ASCII/Unicode。
- 字符串清洗。
- 日期时间格式转换。
- normalization / denormalization。
- 过滤无效记录。
- join lookup tables。
- de-duplication。
- 聚合。
- 生成 surrogate keys。
- 维护 production key 到 DW key 的 mapping table。
- 正确处理历史，尤其是 SCD。

### 4.6 Load

Load 的目标：快速加载到 DW。

注意点：

- Delta load 通常比 total load 快很多。
- 大量 SQL row-by-row update 会很慢。
- DB load tools 往往更快，例如 MySQL `LOAD DATA`。
- Index 会显著拖慢加载，常见做法是先 drop index，加载后 rebuild。
- 可以 parallelization，例如 dimensions 并行、fact tables 并行、partitions 并行。

加载顺序：

1. 先加载/更新 dimensions。
2. 再加载 facts。

原因：fact table 的 foreign keys 必须能找到对应 dimension rows。

### 4.7 Initial Load 与 Incremental Update

Initial load:

- DW 第一次启动时加载全部历史数据。
- 数据量大，很重。
- 历史数据可能不完整或难以修正。

Incremental update:

- DW 运行后周期性加载变化数据。
- 可以按 month/week/day/hour。
- 数据量较小。
- 需要识别 delta。

### 4.8 ETL 示例逻辑

Sakila rental fact table 示例 grain：

- date, store, film_category, number of rental。

准备 dimension tables：

- `DimDate`
- `DimStore`
- `DimCategory`

加载 fact table 的逻辑：

1. 从 rental 抽取 rental date。
2. join inventory 找 store。
3. join film 和 film_category 找 category。
4. join DW 的 dimcategory 找 surrogate key。
5. 按 date、store、category 分组。
6. `COUNT(1)` 得到 rentals 数量。
7. 插入 fact table。

这个例子体现了 ETL 代码题/设计题的核心：先定 grain，再准备 dimensions，再把 source natural keys 转成 DW surrogate keys，最后按 grain 聚合加载 facts。
