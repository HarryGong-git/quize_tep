Schema Design 
’éĘ Most data warehouses use a star schema to represent multi-dimensional 
model. 
’éĘ Each dimension is represented by a dimension table that describes it. 
’éĘ A fact table connects to all dimension tables with a multiple join. Each tuple 
in the fact table consists of a pointer to each of the dimension tables that 
provide its multi-dimensional coordinates and stores measures for those 
coordinates. 
’éĘ The links between the fact table in the center and the dimension tables in the 
extremities form a shape like a star.

The ŌĆ£ClassicŌĆØ Star Schema 
’üĘ A relational model with a one-to-many relationship between dimension table and fact 
table. 
’üĘ A single fact table, with detail and summary data 
’üĘ Fact table primary key has only one key column per dimension 
’üĘ Each dimension is a single table, highly denormalized 
’éĘ Benefits: Easy to understand, intuitive mapping between the business entities, easy to define 
hierarchies, reduces # of physical joins, low maintenance, very simple metadata 
’éĘ Drawbacks: Summary data in the fact table yields poorer performance for summary levels, huge 
dimension tables a problem

What is the Best Design? 
’éĘ Performance benchmarking can be used to determine what is the 
best design. 
’éĘ Snowflake schema: easier to maintain dimension tables when 
dimension tables are very large (reduce overall space). It is not 
generally recommended in a data warehouse environment. 
’éĘ Star schema: more effective for data cube browsing (less joins): can 
affect performance. Recommended design.

Identify the Grain 
’éĘ We must specify exactly what a fact table record means. The grain conveys 
the level of detail associated with the fact table measurements.  
’éĘ Identifying the grain means deciding on the level of detail you want to be 
made available in the dimensional model. 
’éĘ The level of detail available in a star schema (dimensional model) is referred 
to as the grain. 
’éĘ Each fact and dimension table is said to have its own grain or granularity. 
 
’éĘ Choosing the right grain is the most important step for designing the 
dimensional model.

Handle Multiple Grains 
’éĘ There can be more than one grain definition associated with a single business 
process.  
’éĘ Separate fact tables with separate grains should be used to handle such situation. 
’ĆŁ Example: daily, monthly, and yearly tables 
’éĘ Consider the amount of data, space, and the performance requirements when you 
decide how to handle multiple granularities. 
 
’éĘ It is possible to have multiple grains in one fact table, by adding a column called 
the granularity flag. Such a column would indicate whether the data or row in the 
fact table is at the daily, monthly, quarterly, or yearly level.  
’ĆŁ However, this is not a good design.
