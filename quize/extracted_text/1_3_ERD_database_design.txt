ER Diagram (ERD) 
• ER model: entity–relationship model 
• Describes interrelated things of interest in a specific domain 
• Composed of entities (which classify the things of interest) and relationships that can exist 
between instances of those entities 
• Very useful for: 
– Data modelling, design new database 
– Data understanding, figure out connection between tables in existing database 
– Communication tool, align understanding of the domain

--- PAGE 9 ---
Convert Entity Relationship Diagram to Tables 
• Create table for each entity, and consider 
– Primary key 
– Complex attributes such as name,  phone numbers 
– Data type of attributes (e.g., number, text) 
• Create table for relationship, and consider 
– Primary key 
– Or use pair of foreign keys as primary key 
– Data type of attributes 
• Create constraints on the tables 
– E.g., does the column allow missing value? Does the value have to be from a fixed set (e.g., gender) 
 
 
• Both entity and relationship become tables in Relational model

One-to-Many (1:N), Many-to-Many (M:N) 
• One-to-Many (1:N): 
– A single record in one table can be related to multiple records in another table, but each of 
those records relates back to only one record in the first table. 
– Example: One customer can place many orders, but each order belongs to one customer. 
 
• Many-to-Many (M:N): 
– Multiple records in one table can be related to multiple records in another table. This is 
usually implemented using a junction (bridge) table that stores foreign keys from both tables. 
– Example: Students can enroll in many courses, and each course can have many students. 
 
• Many-to-One (N:1): 
– Multiple records in one table are related to a single record in another table. It is essentially the 
reverse of a one-to-many relationship. 
– Example: Many orders belong to one customer.
