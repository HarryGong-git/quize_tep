ETL (in Real World) is Challenging 
• Developing the ETL system is challenging because so many outside constraints put 
pressure on its design:  
– the business requirements,  
– source data realities,  
– budget,  
– processing windows,  
– skill sets of the available staff.  
• Yet it can be hard to appreciate just why the ETL system is so complex and 
resource-intensive. 
 
• A lot of freedom in the design, tools, and implementation

--- PAGE 6 ---
High Level Requirements (Constraints) 
• Data Latency 
– how quickly source system data must be delivered to the business users via the 
DW/BI system 
• Archiving and Lineage 
– staging the data (writing it to disk) after each major activity of the ETL pipeline 
• BI Delivery Interfaces 
– the ETL team, working closely with the modeling team, must take responsibility for 
the content and structure of the data that makes the BI applications simple and 
fast 
• Available Skills 
– whether to hand code the ETL system or use a vendor’s ETL package 
• Legacy Licenses 
– major design decisions will be made implicitly by senior management’s insistence 
that you use existing legacy licenses

--- PAGE 7 ---
ETL Tools 
• ETL tools are the equivalent of schema mappings, but are more powerful 
• Pieces of code to take data from a source, convert it into data for the warehouse: 
– import filters – read and convert from data sources 
– data transformations – join, aggregate, filter, convert data 
– de-duplication – finds multiple records referring to the same entity, merges 
them 
– profiling – builds tables, histograms, etc. to summarize data 
– quality management – test against master values, known business rules, 
constraints, etc.

--- PAGE 10 ---
Data Staging Area (DSA) 
• Transit storage for data underway in the ETL process  
– Transformations/cleansing done here  
• No user queries (some do it)  
• Sequential operations (few) on large data volumes  
– Performed by central ETL logic  
– Easily restarted  
– No need for locking, logging, etc.  
– RDBMS or flat files? (DBMS have become better at this)  
• Finished dimensions copied from DSA to relevant marts

--- PAGE 11 ---
Building Dimensions 
• Static dimension table 
– Relatively easy 
– Assignment of keys: production keys to DW keys using mapping table 
– Combination of data sources: find common key 
• Handling dimension changes 
– Find newest DW key for a given production key 
– Table for mapping production keys to DW keys must be updated 
• Load of dimensions 
– Small dimensions: replace 
– Large dimensions: load only changes 
11 

--- PAGE 12 ---
Building Fact Tables 
• Two types of load 
• Initial load 
– ETL for all data up till now 
– Done when DW is started the first time 
– Often problematic to get correct historical data 
– Very heavy - large data volumes 
• Incremental update 
– Move only changes since last load 
– Done periodically (../month/week/day/hour/...) after DW start 
– Less heavy - smaller data volumes 
• Dimensions must be updated before facts 
– The relevant dimension rows for new facts must be in place 
– Special key considerations if initial load must be performed again

--- PAGE 13 ---
E - Extract 
• Goal: fast extract of relevant data 
– Extract from source systems can take a long time 
• Types of extracts: 
– Extract applications (SQL): co-existence with other applications 
– DB unload tools: much faster than SQL-based extracts 
– Extract applications sometimes the only solution 
• Often too time consuming to ETL all data at each load 
– Extracts can take days/weeks 
– Drain on the operational systems 
– Drain on DW systems 
– => Extract/ETL only changes since last load (delta)

--- PAGE 14 ---
T – Data Transformations 
• Data type conversions 
– EBCDIC->ASCII/UniCode 
– String manipulations 
– Date/time format conversions 
• Normalization/denormalization 
– To the desired DW format 
– Depending on source format 
• Building keys 
– Table that maps production keys to surrogate DW keys 
– Observe correct handling of history - especially for total reload 
14 

--- PAGE 15 ---
L - Load 
• Goal: fast loading into DW 
– Loading deltas is much faster than total load 
• SQL-based update is slow 
– Large overhead (optimization, locking, etc.) for every SQL call 
– DB load tools are much faster 
– Some load tools can also perform UPDATEs 
• Index on tables slows load a lot 
– Drop index and rebuild after load 
– Can be done per partition 
• Parallelization 
– Dimensions can be loaded concurrently 
– Fact tables can be loaded concurrently 
– Partitions can be loaded concurrently

--- PAGE 17 ---
A Few Hints on ETL Design 
• Do not try to implement all transformations in one step! 
 
• Do one (or just a few) thing(s) at a time 
– Copy source data to DSA (data staging area) 
– Compute deltas 
• Only if doing incremental load 
• Handle versions and DW keys 
– Versions only if handling slowly changing dimensions 
• Implement complex transformations 
• Load dimensions 
• Load facts

ETL vs. ELT 
• ETL (Extract, Transform, Load) 
– Traditional approach. Data is gathered from sources, sent to a staging area where it is cleaned and 
formatted, and then pushed into the destination (like a data warehouse). 
• ELT (Extract, Load, Transform) 
– Modern approach. Data is extracted and immediately dumped into a high-powered cloud warehouse (like 
Snowflake or BigQuery). The transformation happens inside the warehouse using its own massive 
computing power.

Compare ETL and ELT 
ETL ELT 
Order Extract -> Transform -> Load Extract -> Load -> Transform 
Complexity High (requires dedicated tools) Low (leverages SQL/Cloud) 
Speed Slower (bottleneck at staging) Faster (parallel processing) 
Cost High for hardware Low (pay-as-you-go cloud) 
Maintenance High (rigid pipelines) Low (flexible/agile)

--- PAGE 26 ---
ELT is getting more popular for new systems 
(especially cloud-based) 
• Cloud warehouses changed the game 
Platforms like Snowflake, Google BigQuery, and Amazon Redshift have massive built-in 
compute, so it makes sense to transform inside them.  
• Storage is cheap 
Keeping raw data is affordable, so teams load first and decide transformations later.  
• Analytics teams want flexibility 
Data analysts and analytics engineers can transform data with SQL later instead of waiting for 
upstream ETL pipelines.  
• Faster iteration 
ELT supports agile experimentation—very useful for startups and modern data teams.
