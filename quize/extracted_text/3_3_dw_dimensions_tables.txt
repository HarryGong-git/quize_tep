Use Surrogate keys as Primary keys for dimension 
tables 
´éÀ It is important that primary keys of dimension tables remain stable.  
´éÀ It is strongly recommended that surrogate keys are created and used for 
primary keys for all dimension tables. 
 
´éÀ Surrogate keys are keys that are maintained within the data warehouse 
instead of the natural keys taken from source data systems. 
´Ç¡ Also known as "dummy keys", "non-natural keys", "artificial keys", 
"meaningless keys", ... 
 
4 

--- PAGE 5 ---
Why Use Surrogate keys 
´éÀ Data tables in various OLTP source systems may use different keys for the same entity. 
´Ç¡ E.g., one source system uses NRIC, another source system uses Passport number 
 
´éÀ It is also possible that a single key is being used by different instances of the same entity 
 
´éÀ This can be a major problem when trying to consolidate information from various source 
systems 
 
´éÀ A surrogate key uniquely identifies each entity in the dimension table, regardless of its 
natural source key.  
 
´éÀ A surrogate key generates a simple integer value for every new entity. 
 
´éÀ Surrogate keys provide the means to maintain data warehouse information when dimensions 
change.

Attributes in Dimension Table 
´éÀ Non-key columns are generally referred to as attributes. 
´éÀ The dimension tables should contain business descriptive columns that users 
reference to define their restriction criteria for ad hoc business queries. 
´éÀ The dimension table attributes show up as report labels inside the reports for 
senior management. 
´éÀ Use a separate field in the dimension table to preserve the natural source system 
key of the entity being used in the source system. 
´éÀ Attributes reflect the potential areas of interest and attributes that can be used 
for aggregations as well as for selective constraints and report breaks. 
´éÀ If a dimension table includes a code, in most cases, include the code description as 
well. 
´éÀ The dimensional attributes serve as headings of the columns of the report and 
should be descriptive and easy to understand. 
´Ç¡ E.g., 0/1 or Y/N is better represented as Yes/No

Date Dimension 
´éÀ The date dimension does not have an OLTP source system connected to it.  
´éÀ The date dimension can be built independently even before the actual dimensional 
design has started.  
´éÀ The best way to built a date dimension is to identify all columns needed and then use 
SQL language to populate the date dimension table for columns such as date, day, 
month, and year.  
´éÀ For other columns, such as holidays, with Christmas or Boxing Day, manual entries may 
be required.  
´éÀ The same is true for columns such as fiscal date, fiscal month, fiscal quarter, and fiscal 
years.  
´éÀ Can also use the spreadsheet to create the date dimension.

Time Dimension 
´éÀ The time dimension is handled separately from the date dimension. 
´éÀ If we have time as part of date dimension, and capture time at second level, 
we would have a huge number of rows in the dimension table even for a 
year: 
365 (Days) x 24 (Hours) x 60 (Minutes) x 60 (Seconds) = 31,536,000 rows 
´éÀ Design the time dimension based on user requirement. 
´Ç¡ hour 
´Ç¡ hour range 
´éÀ Early morning (6 a.m. to 8 a.m.) 
´éÀ Late morning hours (8 a.m. to 11 a.m.) 
´éÀ Rush hour (11 a.m. to 1 p.m.) 
´éÀ Lunch hour (1 p.m. to 2 p.m.)

´éÀ degenerate dimensions: dimensions without any attributes  
´éÀ They are not typical dimensions, but often simply a transaction number that is placed inside the fact table

´Ç¡ transaction numbers 
´Ç¡ bill numbers 
´Ç¡ courier tracking number 
´Ç¡ order number 
´Ç¡ invoice number

Fast changing dimensions 
´éÀ A fast changing dimension is a dimension whose attributes for a record (row) change 
fast over time. Also called rapidly changing dimensions. 
 
´éÀ Cannot be handled using the Type-1, Type-2, or Type-3 approaches used for slowly 
changing dimensions. 
 
´éÀ The best approach for handling very fast changing dimensions is to separate the fast 
changing attributes into one or more separate dimensions which are called mini-
dimensions. 
 
´éÀ The fact table then has two or more foreign keysÔÇöone for the primary dimension table 
and another for the one or more mini-dimensions (consisting of fast changing 
attributes).
