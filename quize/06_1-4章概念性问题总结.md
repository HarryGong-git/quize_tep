# 1-4章概念性问题总结（考试速背版）

使用方法：先看“问题”，自己用 3-5 句话口述；再用“高分答题要点”补全；最后背“一句话记忆版”。

## Chapter 1: Database / OLTP / Relational Model / ERD / Normalization

### 1. 什么是 data？什么是 information？

高分答题要点：

- Data 是关于事物和事件的 raw facts。
- Information 是经过整理、转换、汇总后，对 decision making 有价值的数据。
- Data 更原始、更细；information 更有业务含义。

一句话记忆版：Data 是原材料，information 是能帮助决策的加工结果。

### 2. 什么是 OLTP？

高分答题要点：

- OLTP = Online Transaction Processing。
- 用于支持日常业务交易，例如支付、订单、ATM 取款。
- 特点是大量短小快速的 insert/update/query。
- 数据库通常 highly normalized，以保证一致性和更新效率。

一句话记忆版：OLTP 是支撑日常业务交易的数据库系统。

### 3. 什么是 transaction？为什么重要？

高分答题要点：

- Transaction 是一组作为一个 unit of work 执行的数据库操作。
- 成功则 `COMMIT`，失败则 `ROLLBACK`。
- 它保证并发和故障情况下数据仍然可靠、一致。

一句话记忆版：Transaction 让一组数据库操作要么全成功，要么全失败。

### 4. 什么是 DBMS？

高分答题要点：

- DBMS = Database Management System。
- 它支持数据采集、存储、维护、查询、恢复和安全控制。
- 企业级 DBMS 还提供 concurrency control 和 recovery。

一句话记忆版：DBMS 是管理数据库的软件系统。

### 5. 什么是 relational model？

高分答题要点：

- Relational model 把数据库表示为一组 tables / relations。
- 表由 rows / tuples 和 columns / attributes 组成。
- 它包括数据结构、integrity constraints 和 data manipulation。
- SQL 查询结果也可以看作一张表。

一句话记忆版：Relational model 用表来组织和操作数据。

### 6. 什么是 primary key？

高分答题要点：

- Primary key 是唯一标识表中每一行的列或列组合。
- 它必须 unique 且 not null。
- 它保证 entity integrity，让每个实体可被区分和追踪。

一句话记忆版：Primary key 是一张表里每行记录的唯一身份证。

### 7. 什么是 foreign key？

高分答题要点：

- Foreign key 是一个表中引用另一个表 key 的列或列组合。
- 它通常引用 parent table 的 primary key。
- 它保证 referential integrity，避免无效引用。

一句话记忆版：Foreign key 是表与表之间建立关系的引用键。

### 8. Primary key 和 foreign key 有什么区别？

高分答题要点：

- Primary key 标识本表记录；foreign key 引用其他表记录。
- Primary key 必须 unique and not null；foreign key 可以重复，是否允许 null 取决于业务规则。
- Primary key 保证 entity integrity；foreign key 保证 referential integrity。

一句话记忆版：Primary key 管“我是谁”，foreign key 管“我关联谁”。

### 9. 什么是 integrity constraints？

高分答题要点：

- Entity integrity：primary key 必须唯一且非空。
- Referential integrity：foreign key 必须引用存在的记录。
- Domain integrity：列值必须满足类型、范围、格式或枚举要求。
- User-defined integrity：业务自定义规则，例如取款金额不能超过余额。

一句话记忆版：Integrity constraints 是保证数据库数据正确性的规则。

### 10. 什么是 ERD？有什么用？

高分答题要点：

- ERD = Entity-Relationship Diagram。
- 它描述业务领域中的 entities、attributes 和 relationships。
- 用于数据库设计、理解已有表关系和业务沟通。

一句话记忆版：ERD 是把业务对象和关系画成数据库设计蓝图。

### 11. 如何把 ERD 转成 relational tables？

高分答题要点：

- 每个 entity 通常转成一张 table。
- 为每张表确定 primary key 和 attributes。
- 1:N 关系通常在 N 端放 foreign key。
- M:N 关系通常建立 junction / bridge table。
- 最后添加 `NOT NULL`、`UNIQUE`、`CHECK`、`FOREIGN KEY` 等约束。

一句话记忆版：Entity 变表，relationship 变 foreign key 或 bridge table。

### 12. 什么是 normalization？1NF、2NF、3NF 分别是什么？

高分答题要点：

- Normalization 的目标是减少 redundancy，提高 integrity。
- 1NF：每个 cell 是 atomic value，每行唯一。
- 2NF：非 key 属性依赖整个 key，没有 partial dependency。
- 3NF：非 key 属性只依赖 key，没有 transitive dependency。

一句话记忆版：Normalization 就是让数据少重复、属性只依赖该依赖的 key。

## Chapter 2: SQL Concepts

### 1. SQL 的主要语句类型有哪些？

高分答题要点：

- DDL / definitional：`CREATE TABLE`、`CREATE VIEW`。
- DML / manipulation：`SELECT`、`INSERT`、`UPDATE`、`DELETE`。
- Transaction control：`COMMIT`、`ROLLBACK`。
- Access control：`GRANT`、`REVOKE`。

一句话记忆版：SQL 既能定义结构，也能查询修改数据，还能控制事务和权限。

### 2. 为什么金额类字段应该用 DECIMAL 而不是 FLOAT？

高分答题要点：

- `FLOAT` 是二进制近似存储，可能产生 rounding errors。
- `DECIMAL(W,R)` 是精确十进制表示。
- 金额、会计、收入等需要精确值的字段应使用 `DECIMAL`。

一句话记忆版：钱要精确，所以用 `DECIMAL`，不要用 `FLOAT`。

### 3. CREATE TABLE 中 constraints 的作用是什么？

高分答题要点：

- Constraints 用来保证数据质量和完整性。
- `PRIMARY KEY` 保证唯一且非空。
- `FOREIGN KEY` 保证引用有效。
- `UNIQUE` 防止重复。
- `NOT NULL` 防止缺失。
- `CHECK` 限制取值范围或枚举。

一句话记忆版：Constraints 是建表时写下的数据规则。

### 4. 什么是 SELECT 查询？

高分答题要点：

- `SELECT` 用于从数据库中检索数据。
- `FROM` 指定数据来自哪些表。
- `WHERE` 过滤 individual rows。
- `ORDER BY` 排序，`LIMIT` 限制返回行数，`DISTINCT` 去重。

一句话记忆版：SELECT 是从表中取出符合条件的数据。

### 5. 什么是 JOIN？

高分答题要点：

- JOIN 用于基于相关列组合两个或多个表的 rows。
- 最常见 join condition 是 primary key = foreign key。
- 它是 relational database 中多表查询的核心。

一句话记忆版：JOIN 是按 key 把多张表横向拼起来。

### 6. INNER JOIN 和 LEFT JOIN 有什么区别？

高分答题要点：

- `INNER JOIN` 只返回两边都匹配的 records。
- `LEFT JOIN` 返回左表全部 records，右表匹配不到则右表字段为 null。
- 如果题目要求保留“没有匹配记录”的左表对象，通常用 `LEFT JOIN`。

一句话记忆版：INNER 只要匹配，LEFT 保留左边全部。

### 7. 什么是 Cartesian join / cross join？为什么危险？

高分答题要点：

- Cartesian join 没有 join condition。
- 它返回两张表所有 rows 的组合。
- 结果数量可能爆炸，通常是忘写 join condition 导致的错误。

一句话记忆版：没有连接条件的 join 会把所有行两两组合，非常危险。

### 8. 什么是 SQL aggregation？

高分答题要点：

- Aggregation 是对一组 rows 计算 summary value。
- 常见函数包括 `COUNT`、`SUM`、`AVG`、`MIN`、`MAX`。
- 常用于 total amount、average salary、number of transactions 等问题。

一句话记忆版：Aggregation 是把多行数据汇总成统计结果。

### 9. GROUP BY 的作用是什么？

高分答题要点：

- `GROUP BY` 按指定列把 rows 分成 groups。
- 每个 group 输出一行 summary result。
- `SELECT` 中非聚合列通常必须出现在 `GROUP BY` 中。

一句话记忆版：GROUP BY 决定“按什么维度汇总”。

### 10. WHERE 和 HAVING 有什么区别？

高分答题要点：

- `WHERE` 在 grouping 前过滤 individual rows。
- `HAVING` 在 grouping 后过滤 aggregated groups。
- 涉及 aggregate function 的条件通常放在 `HAVING`。

一句话记忆版：WHERE 筛行，HAVING 筛组。

### 11. JOIN operation 和 SET operation 有什么区别？

高分答题要点：

- JOIN works on columns，是横向合并表。
- SET operation works on rows，是纵向处理结果集。
- `UNION`、`INTERSECT`、`MINUS/EXCEPT` 要求两边列数相同、对应类型兼容。

一句话记忆版：JOIN 横向拼列，SET 纵向拼行。

### 12. 什么是 subquery 和 CTE？

高分答题要点：

- Subquery 是嵌套在另一个 query 中的 query。
- 可用于计算中间结果，例如平均值、最大值、过滤集合。
- CTE = Common Table Expression，用 `WITH` 定义临时命名结果集。
- CTE 可读性更强，适合多步复杂查询。

一句话记忆版：Subquery 是嵌套查询，CTE 是给中间查询起名字。

## Chapter 3: Data Warehouse / OLAP / Dimensional Model

### 1. 为什么不能总是直接在 OLTP 数据库上做 BI？

高分答题要点：

- OLTP 要支持日常业务，复杂分析查询可能影响业务系统性能。
- OLTP 通常 highly normalized，分析时需要 join 很多表。
- 分析需要大量 historical data，会增加 OLTP 负担。
- 因此需要 Data Warehouse 专门支持查询和分析。

一句话记忆版：OLTP 负责跑业务，DW/OLAP 负责做分析。

### 2. 什么是 Data Warehouse？

高分答题要点：

- Data Warehouse 是 integrated、subject-oriented 的数据仓库。
- 它保存 historical data，通常 non-volatile，更新不频繁。
- 它为 OLAP、BI dashboard 和 decision support 提供数据基础。
- 它面向分析优化，而不是面向交易优化。

一句话记忆版：Data Warehouse 是为历史分析和决策支持准备的数据仓库。

### 3. “Data warehouse has less data because it only contains summarized data” 对吗？

高分答题要点：

- 不对，或至少过度简化。
- DW 不只包含 summarized data，也可以包含 detailed historical data。
- DW 可能整合多个 source systems，并保存长时间历史，所以可能比单个 operational database 更大。
- DW 可以有 summary / aggregate tables，但这不是它唯一的数据。

一句话记忆版：DW 可以有汇总数据，但不等于只存汇总数据。

### 4. OLTP 和 OLAP 有什么区别？

高分答题要点：

- OLTP 支持 daily operations；OLAP 支持 analysis and decision making。
- OLTP mostly updates；OLAP mostly reads。
- OLTP normalized；OLAP often denormalized。
- OLTP 保存 current snapshot；OLAP 保存 full history。
- OLTP 查询短小简单；OLAP 查询复杂且跨大量数据。

一句话记忆版：OLTP 管交易，OLAP 管分析。

### 5. 什么是 Data Mart？

高分答题要点：

- Data Mart 是 Data Warehouse 的子集。
- 它通常面向某个部门、团队或业务线。
- 优点是范围小、上线快；缺点是长期跨部门整合可能复杂。

一句话记忆版：Data Mart 是面向局部业务的“小数据仓库”。

### 6. 什么是 Dimensional Model？

高分答题要点：

- Dimensional modeling 围绕业务概念组织数据。
- 它关注 measures 和 dimensions。
- 相比 ER model 的 entities and relationships，dimensional model 更适合分析和报表。

一句话记忆版：Dimensional model 用“指标 + 维度”来组织分析数据。

### 7. 什么是 measure 和 dimension？

高分答题要点：

- Measure 是可分析的数值指标，例如 sales amount、quantity、number of rentals。
- Dimension 是描述 measure 上下文的业务参数，例如 time、customer、product、store。
- 常见分析形式是 measure by dimensions。

一句话记忆版：Measure 是算什么，dimension 是按什么看。

### 8. 什么是 fact table 和 dimension table？

高分答题要点：

- Fact table 保存业务过程的 measurements。
- Fact table 包含 measures 和指向 dimension tables 的 foreign keys。
- Dimension table 保存描述性 attributes，用于筛选、分组、汇总和报表标签。
- Fact table 的 grain 决定每行代表的业务细节。

一句话记忆版：Fact table 存指标，dimension table 存描述。

### 9. Star schema 和 snowflake schema 有什么区别？

高分答题要点：

- Star schema：一个 fact table 在中心，多个 denormalized dimension tables 围绕它。
- Star schema 直观、join 少、查询性能好，通常更适合 DW/OLAP。
- Snowflake schema：把 dimension tables 进一步 normalized。
- Snowflake 减少冗余、便于维护，但 join 更多、查询更复杂。

一句话记忆版：Star 快而直观，snowflake 更规范但更复杂。

### 10. 什么是 grain？为什么重要？

高分答题要点：

- Grain 定义 fact table 每一行代表的详细程度。
- 例如一张收据的一条 line item，或每天每店每类别的租赁数量。
- Grain 决定 measures、dimension keys 和可分析粒度。
- 定义 grain 是 dimensional model 最重要的步骤。

一句话记忆版：Grain 说明 fact table 的“一行到底代表什么”。

### 11. 为什么 dimension table 推荐使用 surrogate key？

高分答题要点：

- Surrogate key 是 DW 内部生成的 artificial key。
- 它比 source natural key 更稳定。
- 它能整合多个 source system 中不同的 key。
- 它支持 dimension change 和历史追踪，尤其是 SCD Type 2。

一句话记忆版：Surrogate key 是 DW 自己给维度行发的稳定编号。

### 12. 什么是 Slowly Changing Dimension？

高分答题要点：

- SCD 指 attributes 会随时间缓慢变化的 dimension。
- Type 1：覆盖旧值，不保留历史。
- Type 2：新增一行和新 surrogate key，保留历史。
- Type 3：增加 previous value 列，保留有限历史。

一句话记忆版：SCD 解决维度属性变化后历史怎么保留的问题。

## Chapter 4: ETL

### 1. 什么是 ETL？

高分答题要点：

- ETL = Extract, Transform, Load。
- Extract 从 source systems 抽取数据。
- Transform 清洗、转换、整合、去重、生成 key、聚合。
- Load 把处理后的数据加载到 Data Warehouse。

一句话记忆版：ETL 是把源系统数据搬进数据仓库并整理好的过程。

### 2. 什么是 Data Staging Area？

高分答题要点：

- Data Staging Area, DSA, 是 ETL 中的中转存储区域。
- 它暂存 extracted data，并执行 cleaning 和 transformation。
- 通常不面向最终用户查询。

一句话记忆版：DSA 是 ETL 路上的临时加工区。

### 3. 为什么 ETL 需要 Data Staging Area？

高分答题要点：

- ETL 处理大数据量和复杂转换，需要分阶段执行。
- DSA 让流程更容易 restart、debug、audit。
- 可以保存 intermediate results、lineage 和 data quality checks。
- 它减少 source systems 和 final DW tables 被复杂转换直接影响的风险。

一句话记忆版：DSA 让 ETL 更可控、可追踪、可恢复。

### 4. Extract 阶段做什么？

高分答题要点：

- Extract 从 source systems 快速抽取 relevant data。
- 可以使用 SQL extracts、DB unload tools 或 extract applications。
- 大型系统通常不每次全量抽取，而是抽取 delta。

一句话记忆版：Extract 是从源系统取出需要的数据。

### 5. Transform 阶段做什么？

高分答题要点：

- Transform 把 source data 转成 DW 需要的格式和语义。
- 包括数据类型转换、日期格式转换、字符串清洗、过滤无效记录。
- 还包括 de-duplication、join lookup、aggregation、生成 surrogate keys。
- 对维度变化还要正确处理历史。

一句话记忆版：Transform 是把脏乱源数据清洗整合成可分析数据。

### 6. Load 阶段做什么？

高分答题要点：

- Load 把转换后的数据加载进 DW。
- 目标是快速、可靠地写入 dimension tables 和 fact tables。
- 大量加载时可使用 load tools，并注意 indexes 对加载速度的影响。

一句话记忆版：Load 是把加工好的数据写进数据仓库。

### 7. Initial load 和 incremental update 有什么区别？

高分答题要点：

- Initial load 是 DW 第一次建立时加载全部历史数据。
- 它数据量大、耗时长，也可能遇到历史数据质量问题。
- Incremental update 是 DW 运行后周期性加载变化数据。
- Incremental update 数据量较小，但需要识别 delta。

一句话记忆版：Initial load 装历史全量，incremental update 装后续变化。

### 8. 为什么要先加载 dimensions，再加载 facts？

高分答题要点：

- Fact table 包含指向 dimension tables 的 foreign keys。
- 新 fact rows 需要找到对应 dimension rows 和 surrogate keys。
- 如果 dimension 不存在，fact 的 referential integrity 会失败。

一句话记忆版：Fact 要引用 dimension，所以 dimension 必须先准备好。

### 9. ETL 设计会受哪些现实约束影响？

高分答题要点：

- Business needs 和 KPIs。
- Compliance、security、data quality。
- Data latency，即数据多快到达业务用户。
- Data integration，例如 conformed dimensions and facts。
- Archiving、lineage、available skills 和 legacy licenses。

一句话记忆版：ETL 设计不仅看技术，还受业务、合规、质量、时效和资源限制。

### 10. 构建 fact table 的 ETL 主逻辑是什么？

高分答题要点：

- 先定义 fact table 的 grain。
- 从 source tables 抽取与 grain 对应的数据。
- Join dimension tables，把 source natural keys 转成 DW surrogate keys。
- 按 grain 聚合 measures。
- Load 到 fact table。

一句话记忆版：建 fact table 先定 grain，再找维度 key，最后按粒度汇总加载。
